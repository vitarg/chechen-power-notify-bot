# Telegram-бот уведомлений о плановых отключениях электричества

## Цель

Разработать Telegram-бота, который уведомляет пользователей о плановых отключениях электричества на основе публикаций Чеченэнерго.

Пользователь добавляет один или несколько адресов. Бот регулярно проверяет сайт Чеченэнерго, сохраняет найденные плановые отключения в базу данных, сопоставляет зоны отключений с адресами пользователей и отправляет релевантные уведомления.

## Источники данных

### MVP

- Основной источник: сайт Чеченэнерго.
- Точный URL раздела с плановыми отключениями будет добавлен позже через конфигурацию.
- Telegram-канал Чеченэнерго в MVP не используется.

### Будущее развитие

- Telegram-канал можно добавить как резервный источник или источник сверки.
- Приоритет на старте: надежный парсинг сайта.

## Расписание проверок

- Проверять сайт каждый час.
- Окно проверок: с 10:00 до 18:00.
- Проверки выполняются в рабочие дни.
- Часовой пояс: `Europe/Moscow`.
- У администратора должна быть команда для ручного запуска синхронизации.

## Пользовательский сценарий

### Первый запуск

1. Пользователь запускает бота командой `/start`.
2. Бот просит ввести адрес свободным текстом.
3. Пользователь вводит адрес, например: `Грозный, ул. Мира`.
4. Бот пытается распознать:
   - населенный пункт;
   - улицу, микрорайон или ориентир, если они есть;
   - исходный текст адреса.
5. Бот показывает пользователю, как понял адрес, и просит подтвердить.
6. Если адрес распознан неуверенно или слишком общий, бот предупреждает пользователя.

Пример:

```text
Я понял адрес так:
Населенный пункт: Грозный
Улица: не распознана

Так можно сохранить, но вы будете получать уведомления по всему Грозному.
Уточнить улицу?
```

Кнопки:

- `Уточнить`
- `Сохранить так`

### Несколько адресов

Один пользователь может хранить несколько адресов.

Каждый адрес должен иметь пользовательское название, например:

- `Дом`
- `Работа`
- `Родители`

Если одно отключение затрагивает несколько адресов одного пользователя, бот отправляет одно сообщение с перечислением затронутых адресов.

## Пользовательские команды MVP

```text
/start - регистрация и добавление первого адреса
/addresses - управление адресами
/add_address - добавить адрес
/history - последние уведомления
/help - помощь
/delete_me - удалить пользователя, адреса и историю уведомлений
```

Редактирование и удаление отдельных адресов выполняется через кнопки внутри `/addresses`.

## Сопоставление адресов

Сопоставление должно учитывать стиль публикаций Чеченэнерго.

Публикации могут быть разной детализации:

- весь населенный пункт;
- часть населенного пункта без конкретных улиц;
- конкретные улицы;
- микрорайоны или ориентиры.

Адрес пользователя нужно хранить не только как сырой текст, но и как разобранные поля:

- исходный адрес;
- нормализованный адрес;
- населенный пункт;
- улица, если есть;
- микрорайон или ориентир, если есть;
- уровень уверенности распознавания.

Правила MVP:

- Если отключение указано по конкретной улице, уведомлять пользователей с совпавшим населенным пунктом и улицей.
- Если отключение указано по всему населенному пункту или части населенного пункта без улиц, уведомлять пользователей с адресами внутри этого населенного пункта.
- Широкие уведомления включены автоматически, отдельной пользовательской настройки в MVP нет.
- Геокодинг и полноценная адресная база в MVP не обязательны.

## Уведомления

### Первичное уведомление

Отправляется сразу после обнаружения нового релевантного отключения.

Пример:

```text
Плановое отключение

Адрес: Дом
Зона: г. Грозный, ул. Мира
Дата: завтра, 12 июня
Время: 09:00-17:00

Причина: ремонтные работы
Источник: Чеченэнерго
```

Если отключение затрагивает несколько адресов пользователя:

```text
Плановое отключение

Затронутые адреса: Дом, Работа
Зона: г. Грозный, ул. Мира, ул. Ленина
Дата: завтра, 12 июня
Время: 09:00-17:00

Причина: ремонтные работы
Источник: Чеченэнерго
```

### Утреннее напоминание

- Отправлять утром в день отключения.
- Время напоминания: 08:00 `Europe/Moscow`.
- Напоминание отправляется только если отключение актуально.

### Несколько отключений

Дайджест в MVP не нужен.

Если найдено несколько разных отключений, бот отправляет отдельное сообщение по каждому отключению.

## История уведомлений

Историю отправленных уведомлений нужно хранить.

Это нужно, чтобы:

- не отправлять одно и то же уведомление повторно;
- различать первичное уведомление и утреннее напоминание;
- показывать пользователю историю по команде `/history`;
- разбирать спорные случаи.

Команда `/history` показывает последние 10 релевантных уведомлений или отключений по адресам пользователя.

## Изменения в публикациях

Если Чеченэнерго изменил уже опубликованное отключение, бот должен обнаружить изменение.

Подход:

- хранить fingerprint публикации;
- хранить версию события отключения;
- при изменении даты, времени или зоны создавать новую версию;
- отправлять затронутым пользователям уведомление об обновлении.

Пример:

```text
Обновление по плановому отключению

Адрес: Дом
Было: 09:00-17:00
Стало: 10:00-16:00
```

Отдельную логику отмены отключений в MVP не делаем.

## Импорт актуальных отключений

При первом запуске парсера нужно импортировать актуальные будущие отключения с сайта.

После добавления нового адреса бот должен сразу проверить уже известные актуальные отключения и отправить пользователю релевантные уведомления, если они есть.

## Админская часть

Админка в MVP реализуется внутри Telegram.

Админские команды:

```text
/sync - вручную запустить проверку источника
/sync_dry_run - проверить парсинг и матчинг без рассылки
/latest - показать последние найденные отключения
/stats - показать пользователей, адреса, уведомления
/broadcast - ручная рассылка для экстренных случаев
/sources - статус последней проверки сайта
```

Администраторы задаются списком Telegram ID в `.env`.

Отдельная веб-админка в MVP не нужна.

## Dry run

Команда `/sync_dry_run` должна:

- запустить парсинг сайта;
- показать найденные отключения;
- показать количество потенциально затронутых пользователей;
- не сохранять изменения, если это удобно реализовать;
- не отправлять уведомления пользователям.

Главная цель: безопасно отлаживать парсинг и сопоставление адресов.

## Ошибки источника

Если сайт Чеченэнерго недоступен или структура страницы изменилась:

- пользователей не уведомлять;
- первую ошибку записать в лог;
- после 2-3 повторных ошибок подряд уведомить администратора.

Пример сообщения админу:

```text
Проблема с источником

Сайт Чеченэнерго недоступен или изменилась структура страницы.
Последняя успешная проверка: 11 июня 2026, 14:00
Ошибка: ...
```

## Хранение исходных публикаций

В базе нужно хранить:

- сырой текст публикации;
- ссылку на источник;
- дату обнаружения;
- нормализованную структуру отключения.

Полный HTML страницы хранить не нужно.

## Персональные данные

Хранить минимально необходимые данные:

- Telegram ID;
- имя и username, если доступны;
- адреса пользователя;
- историю отправленных уведомлений.

Пользователю нужна команда `/delete_me`, которая удаляет:

- профиль пользователя;
- адреса;
- историю уведомлений.

## Срок хранения данных

Старые данные хранить 3 месяца:

- отключения;
- версии отключений;
- историю уведомлений;
- сырые тексты публикаций.

Нужна плановая задача очистки старых данных.

## Технический стек

Оптимальный стек под сервер 2 vCPU, 1 GB RAM, 15 GB SSD:

- Python;
- aiogram;
- PostgreSQL;
- APScheduler внутри процесса бота;
- httpx для HTTP-запросов;
- BeautifulSoup или lxml для парсинга HTML;
- systemd для запуска сервиса;
- без Docker на старте.

Почему без Docker:

- меньше накладных расходов по памяти;
- проще уложиться в 1 GB RAM;
- PostgreSQL можно установить из apt;
- бот можно запускать из Python virtualenv через systemd.

Не использовать в MVP:

- Celery;
- Redis;
- Kubernetes;
- отдельную веб-админку;
- Selenium, если сайт можно парсить обычным HTTP.

## Деплой

Формат деплоя:

- Python virtualenv;
- systemd service для бота;
- PostgreSQL из apt;
- настройки через `.env`.

В `.env` должны быть:

```text
BOT_TOKEN=
DATABASE_URL=
ADMIN_IDS=
SOURCE_URL=
TIMEZONE=Europe/Moscow
```

`SOURCE_URL` будет заполнен позже.

## Бэкапы

Бэкапы базы нужны.

MVP-подход:

- ежедневный `pg_dump`;
- хранить последние 7 ежедневных копий локально на сервере;
- позже можно добавить внешний бэкап.

## Язык интерфейса

Интерфейс MVP: русский.

Возможность добавить чеченский язык можно заложить архитектурно, но не реализовывать на старте.

## Что не входит в MVP

- Парсинг Telegram-канала Чеченэнерго.
- Веб-сайт или лендинг.
- Веб-админка.
- Геокодинг.
- Полноценная адресная база.
- Дайджест уведомлений.
- Отслеживание отмен отключений.
- Пользовательская настройка широких уведомлений.
- Docker.
- Celery/Redis.

## Открытые вопросы перед реализацией

1. Указать точный URL страницы Чеченэнерго с плановыми отключениями.
2. Посмотреть реальные публикации и выделить типовые форматы текста.
3. Решить, насколько глубоко нормализовать названия населенных пунктов и улиц на первом этапе.
4. Определить формат таблиц PostgreSQL.
5. Подготовить MVP-план реализации по этапам.
